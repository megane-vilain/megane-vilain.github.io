# Volumes

## Qu'est-ce qu'un volume ?

Un **volume** est un mécanisme permettant de fournir un espace de stockage à un Pod.

Les volumes sont définis au niveau du Pod puis montés dans un ou plusieurs conteneurs.

Ils permettent notamment :

- de partager des données entre plusieurs conteneurs d'un même Pod ;
- de conserver des données lorsqu'un conteneur redémarre ;
- de fournir des fichiers de configuration ;
- de monter des Secrets ou des ConfigMaps ;
- d'utiliser un stockage persistant.

---

## Pourquoi utiliser un volume ?

Par défaut, le système de fichiers d'un conteneur n'est pas conçu pour stocker durablement des données.

Si le conteneur est recréé, les modifications apportées à son système de fichiers peuvent être perdues.

Les volumes permettent de stocker les données à l'extérieur du système de fichiers du conteneur.

---

## Comment fonctionne un volume ?

Un volume est déclaré dans la définition du Pod.

Les conteneurs qui souhaitent y accéder doivent ensuite le monter à l'aide de `volumeMounts`.

```
Pod
│
├── Volume
│
├── Conteneur A
│      │
│      └── /data
│
└── Conteneur B
       │
       └── /shared
```

Les deux conteneurs peuvent ainsi accéder aux mêmes données.

---

## Exemple minimal

```yaml
apiVersion: v1
kind: Pod

metadata:
  name: demo

spec:
  volumes:
    - name: shared-data
      emptyDir: {}

  containers:
    - name: app
      image: busybox

      volumeMounts:
        - name: shared-data
          mountPath: /data

    - name: worker
      image: busybox

      volumeMounts:
        - name: shared-data
          mountPath: /shared
```

Les deux conteneurs partagent le même volume.

# Les principaux types de volumes

Kubernetes propose de nombreux types de volumes.

Les plus courants sont présentés ci-dessous.

---

## emptyDir

Un volume `emptyDir` est créé lorsque le Pod démarre.

Il est partagé entre les conteneurs du Pod.

Toutes les données sont supprimées lorsque le Pod disparaît.

```yaml
volumes:
  - name: cache
    emptyDir: {}
```

Cas d'utilisation :

- fichiers temporaires ;
- cache ;
- échange de données entre conteneurs.

---

## configMap

Une ConfigMap peut être montée sous forme de volume.

Chaque clé devient un fichier.

```yaml
volumes:
  - name: config
    configMap:
      name: app-config
```

Cette méthode est souvent utilisée pour fournir des fichiers de configuration.

---

## secret

Les Secrets peuvent également être montés dans un volume.

```yaml
volumes:
  - name: credentials
    secret:
      secretName: database-secret
```

Cette approche est particulièrement adaptée aux certificats et aux clés privées.

---

## persistentVolumeClaim

Un volume peut également utiliser un **PersistentVolumeClaim (PVC)**.

```yaml
volumes:
  - name: data

    persistentVolumeClaim:
      claimName: app-storage
```

Dans ce cas, le stockage est fourni par un **PersistentVolume**.

Les données persistent même si le Pod est supprimé.

Cette méthode est utilisée pour :

- les bases de données ;
- les applications stockant des fichiers ;
- les applications nécessitant un stockage durable.

---

# Le stockage persistant

Pour stocker durablement des données, Kubernetes utilise généralement trois ressources :

```
Pod
 │
 ▼
PersistentVolumeClaim
 │
 ▼
PersistentVolume
 │
 ▼
Stockage physique
```

Le Pod ne référence jamais directement un PersistentVolume.

Il utilise un PersistentVolumeClaim, qui représente une demande de stockage.

---

## PersistentVolume (PV)

Le **PersistentVolume** représente une ressource de stockage disponible dans le cluster.

Il peut correspondre à :

- un disque local ;
- un partage NFS ;
- un volume cloud ;
- un système de stockage distribué.

Les PersistentVolumes sont généralement créés automatiquement grâce aux StorageClasses.

---

## PersistentVolumeClaim (PVC)

Le **PersistentVolumeClaim** représente une demande de stockage.

Par exemple :

```yaml
resources:
  requests:
    storage: 20Gi
```

Le cluster fournit alors un volume répondant à cette demande.

Le Pod utilise ensuite ce PVC.

---

## StorageClass

La **StorageClass** décrit la manière dont le stockage doit être créé.

Par exemple :

- SSD ;
- HDD ;
- stockage haute performance ;
- stockage réseau.

Lorsqu'un PVC référence une StorageClass, Kubernetes peut créer automatiquement le volume correspondant.

Dans la majorité des clusters modernes, ce provisionnement est automatique.

---

# Volumes et Pods

Les volumes appartiennent toujours au Pod.

Cela signifie que :

- tous les conteneurs du Pod peuvent les partager ;
- si un Pod est recréé, certains volumes disparaissent (`emptyDir`) ;
- d'autres restent disponibles (`PersistentVolumeClaim`).

Le type de volume détermine donc la durée de vie des données.

---

## Choisir le bon type de volume

| Besoin | Type recommandé |
|---------|-----------------|
| Partage entre conteneurs | `emptyDir` |
| Configuration | `configMap` |
| Données sensibles | `secret` |
| Stockage durable | `PersistentVolumeClaim` |

---

## Bonnes pratiques

- Choisir le type de volume adapté au besoin.
- Utiliser un PVC pour les données importantes.
- Éviter `hostPath` en production sauf cas particuliers.
- Monter les Secrets en lecture seule lorsque cela est possible.
- Séparer les données persistantes des fichiers temporaires.

---

## Commandes utiles

Lister les PersistentVolumes :

```bash
kubectl get pv
```

Lister les PersistentVolumeClaims :

```bash
kubectl get pvc
```

Afficher les détails d'un PVC :

```bash
kubectl describe pvc <nom>
```

Afficher les détails d'un PV :

```bash
kubectl describe pv <nom>
```

---

## Ce qu'il faut retenir

- Un volume fournit un espace de stockage à un Pod.
- Les volumes sont définis au niveau du Pod puis montés dans les conteneurs.
- Kubernetes propose plusieurs types de volumes selon les besoins.
- Les données réellement persistantes utilisent généralement un PersistentVolumeClaim.
- Les StorageClasses permettent de créer automatiquement les volumes de stockage.
