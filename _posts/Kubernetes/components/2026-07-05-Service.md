# Services

## Qu'est-ce qu'un Service ?

Un **Service** est une ressource Kubernetes qui permet d'exposer une application sur le réseau.

Les Pods étant éphémères, ils peuvent être recréés à tout moment avec une nouvelle adresse IP. Il n'est donc pas recommandé de communiquer directement avec eux.

Le Service fournit un **point d'accès stable** vers un ou plusieurs Pods.

Les clients communiquent avec le Service, qui redirige ensuite les requêtes vers les Pods correspondants.

---

## Pourquoi utiliser un Service ?

Prenons l'exemple d'une application exécutée dans trois Pods :

```
api-abc123   10.42.0.15
api-def456   10.42.1.08
api-ghi789   10.42.2.31
```

Si l'un de ces Pods est recréé, son adresse IP changera.

Un client utilisant directement ces adresses IP risquerait alors de ne plus pouvoir communiquer avec l'application.

En créant un Service, Kubernetes attribue une adresse IP stable au groupe de Pods.

```
Client
   │
   ▼
Service
   │
   ├── api-abc123
   ├── api-def456
   └── api-ghi789
```

Les Pods peuvent être recréés sans que les clients aient besoin de connaître leurs nouvelles adresses.

---

## Comment un Service trouve les Pods ?

Comme les Deployments, les Services utilisent des **labels** et des **selectors**.

Par exemple, les Pods peuvent posséder le label :

```yaml
labels:
  app: api
```

Le Service sélectionne ensuite tous les Pods possédant ce label.

```yaml
selector:
  app: api
```

Dès qu'un nouveau Pod portant ce label est créé, il est automatiquement ajouté au Service.

À l'inverse, lorsqu'un Pod disparaît, il est automatiquement retiré.

---

## Exemple minimal

```yaml
apiVersion: v1
kind: Service

metadata:
  name: api

spec:
  selector:
    app: api

  ports:
    - port: 80
      targetPort: 8080
```

Dans cet exemple :

- le Service écoute sur le port **80** ;
- les requêtes sont redirigées vers le port **8080** des Pods ;
- tous les Pods possédant le label `app: api` sont utilisés.

---

## Les principaux champs

### selector

Le selector indique quels Pods reçoivent les requêtes.

```yaml
selector:
  app: api
```

Les labels doivent correspondre à ceux définis dans les Pods.

---

### ports

Cette section décrit la correspondance entre le port du Service et celui des Pods.

```yaml
ports:
  - port: 80
    targetPort: 8080
```

Dans cet exemple :

- les clients communiquent avec le Service sur le port **80** ;
- Kubernetes redirige ensuite les requêtes vers le port **8080** des Pods.

---

### type

Le champ `type` détermine la manière dont le Service est exposé.

Si ce champ est omis, Kubernetes crée un Service de type **ClusterIP**.

---

## Les différents types de Services

### ClusterIP

C'est le type par défaut.

Le Service est accessible uniquement depuis le cluster Kubernetes.

```
Application A
      │
      ▼
ClusterIP Service
      │
      ▼
Pods
```

C'est le type utilisé pour la communication entre applications.

---

### NodePort

Le Service est exposé sur un port de chaque Worker Node.

Les clients peuvent accéder au Service en utilisant :

```
IP du Worker Node : NodePort
```

Ce type est principalement utilisé pour les environnements de test ou de développement.

---

### LoadBalancer

Lorsque le cluster est connecté à un fournisseur de cloud ou à un équilibreur de charge compatible, Kubernetes peut créer automatiquement un Load Balancer.

```
Internet
    │
    ▼
Load Balancer
    │
    ▼
Service
    │
    ▼
Pods
```

C'est le mode généralement utilisé pour exposer une application vers l'extérieur.

---

### Headless Service

Un **Headless Service** est un Service qui ne possède pas d'adresse IP virtuelle.

Il permet aux clients de communiquer directement avec chaque Pod.

Ce type est principalement utilisé avec les **StatefulSets**, où chaque Pod possède une identité propre.

---

## Répartition de charge

Lorsqu'un Service cible plusieurs Pods, Kubernetes répartit automatiquement les requêtes entre eux.

Par exemple :

```
                Service
              /    |    \
             /     |     \
        Pod A   Pod B   Pod C
```

Cette répartition permet d'utiliser plusieurs instances d'une même application.

Le mécanisme exact dépend de l'implémentation réseau du cluster.

---

## DNS

Chaque Service reçoit automatiquement un nom DNS dans le cluster.

Par exemple, un Service nommé :

```
api
```

pourra généralement être atteint par :

```
http://api
```

Les applications n'ont donc pas besoin de connaître les adresses IP des Pods.

---

## Ce que le Service ne fait pas

Un Service ne crée pas de Pods.

Il ne remplace pas un Deployment.

Son rôle est uniquement de fournir un point d'accès stable vers des Pods existants.

De même, un Service ne publie pas automatiquement une application sur Internet.

Pour exposer une application HTTP ou HTTPS à l'extérieur du cluster, on utilise généralement un **Ingress** ou un **Gateway**, selon l'architecture mise en place.

---

## Quand utiliser un Service ?

Un Service est utilisé dès qu'une application doit être accessible par :

- une autre application du cluster ;
- plusieurs Pods ;
- un utilisateur (via un Ingress ou un Load Balancer).

La plupart des Deployments sont donc associés à un Service.

---

## Commandes utiles

Lister les Services :

```bash
kubectl get services
```

Afficher les détails :

```bash
kubectl describe service <nom>
```

Modifie le manifeste :

```bash
kubectl edit service <nom>
```

Supprimer un Service :

```bash
kubectl delete service <nom>
```

---

## Bonnes pratiques

- Utiliser des labels cohérents entre les Pods et les Services.
- Ne jamais communiquer directement avec l'adresse IP d'un Pod.
- Utiliser le type **ClusterIP** par défaut pour les communications internes.
- Réserver les types **NodePort** et **LoadBalancer** aux cas où ils sont réellement nécessaires.
- Utiliser un **Headless Service** avec un StatefulSet lorsque chaque Pod doit être adressable individuellement.

---

## Ce qu'il faut retenir

- Un Service fournit un point d'accès stable vers un groupe de Pods.
- Il sélectionne les Pods grâce aux labels.
- Les Pods peuvent être recréés sans impact pour les clients.
- Un Service peut répartir les requêtes entre plusieurs Pods.
- Le type de Service détermine comment l'application est exposée.
