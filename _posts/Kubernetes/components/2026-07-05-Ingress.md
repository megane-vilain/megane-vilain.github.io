# Ingress

## Qu'est-ce qu'un Ingress ?

Un **Ingress** est une ressource Kubernetes qui permet d'exposer des applications HTTP ou HTTPS vers l'extérieur du cluster.

Il agit comme un **point d'entrée unique** pour plusieurs Services, en fonction de règles définies (souvent basées sur le nom de domaine ou le chemin URL).

Un Ingress ne fonctionne pas seul : il a besoin d'un **Ingress Controller** (comme NGINX Ingress Controller, Traefik, etc.) pour être exécuté.

---

## Pourquoi utiliser un Ingress ?

Sans Ingress, une application peut être exposée de plusieurs façons :

- NodePort (peu flexible) ;
- LoadBalancer (un par service, souvent coûteux ou limité) ;
- accès direct via des IPs de nodes.

Cela devient vite difficile à gérer lorsqu'il y a plusieurs applications.

L'Ingress permet de :

- centraliser l'accès HTTP/HTTPS ;
- router plusieurs applications via un seul point d'entrée ;
- utiliser des noms de domaine ;
- gérer le TLS (HTTPS).

---

## Comment fonctionne un Ingress ?

Un Ingress définit des **règles de routage**.

Ces règles indiquent :

- quel domaine ou chemin doit être utilisé ;
- vers quel Service rediriger le trafic.

Exemple :

```
api.monsite.com  → Service API
app.monsite.com  → Service Frontend
```

ou

```
/api     → Service API
/        → Service Frontend
```

Le trafic arrive d'abord sur l'Ingress Controller, qui applique les règles et transmet la requête au bon Service.

---

## Architecture simplifiée

```
Internet
    │
    ▼
Ingress Controller
    │
    ├── Service API
    └── Service Frontend
            │
            ▼
           Pods
```

---

## Exemple minimal

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress

metadata:
  name: example-ingress

spec:
  rules:
    - host: example.local
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: frontend
                port:
                  number: 80
```

Dans cet exemple :

- toutes les requêtes vers `example.local` sont routées vers le Service `frontend` ;
- le Service redistribue ensuite vers les Pods.

---

## Les principaux champs

### rules

Les règles définissent comment le trafic est routé.

```yaml
rules:
  - host: example.local
```

Le champ `host` permet de router en fonction du nom de domaine.

---

### paths

Les paths permettent de router selon l'URL.

```yaml
paths:
  - path: /api
    pathType: Prefix
```

Cela permet par exemple de séparer plusieurs applications sur un même domaine.

---

### backend

Le backend définit le Service cible.

```yaml
backend:
  service:
    name: api
    port:
      number: 80
```

L'Ingress ne parle jamais directement aux Pods, uniquement aux Services.

---

## Ingress Controller

Un **Ingress Controller** est un composant indispensable.

Il observe les ressources Ingress et configure un proxy HTTP (comme NGINX ou Traefik) en conséquence.

Sans Ingress Controller :

- les ressources Ingress existent dans Kubernetes ;
- mais elles ne font rien.

Exemples d'Ingress Controllers :

- NGINX Ingress Controller ;
- Traefik ;
- HAProxy Ingress ;
- cloud providers (AWS ALB, GCP, Azure).

---

## TLS / HTTPS

Un Ingress peut gérer le HTTPS en utilisant des certificats TLS.

Exemple :

```yaml
tls:
  - hosts:
      - example.local
    secretName: example-tls
```

Le secret contient le certificat et la clé privée.

Cela permet de terminer le TLS directement au niveau de l'Ingress.

---

## Ingress vs Service

| Ressource | Rôle |
|-----------|------|
| Service | Expose les Pods dans le cluster |
| Ingress | Expose les Services vers l'extérieur (HTTP/HTTPS) |

Le Service est interne au cluster, l'Ingress est une couche d'entrée externe.

---

## Quand utiliser un Ingress ?

Utiliser un Ingress lorsque :

- plusieurs applications doivent être exposées via HTTP/HTTPS ;
- un nom de domaine est nécessaire ;
- on veut centraliser l'accès réseau ;
- on veut gérer TLS au même endroit.

---

## Quand ne pas utiliser un Ingress ?

Un Ingress n'est pas adapté pour :

- les protocoles non HTTP (ex: TCP brut, UDP) ;
- les services internes uniquement ;
- les cas simples où un NodePort suffit.

---

## Ce que l'Ingress ne fait pas

Un Ingress :

- ne crée pas de Pods ;
- ne remplace pas un Service ;
- ne fonctionne pas sans Ingress Controller ;
- ne gère pas directement la répartition entre Pods (c'est le Service qui le fait).

---

## Commandes utiles

Lister les Ingress :

```bash
kubectl get ingress
```

Afficher les détails :

```bash
kubectl describe ingress <nom>
```

Supprimer un Ingress :

```bash
kubectl delete ingress <nom>
```

---

## Bonnes pratiques

- Toujours utiliser un Service derrière un Ingress.
- Préférer le routing par hostname plutôt que par path quand possible.
- Centraliser le TLS au niveau de l'Ingress.
- Vérifier quel Ingress Controller est utilisé dans le cluster.
- Éviter de mélanger trop de règles complexes dans un seul Ingress.

---

## Ce qu'il faut retenir

- Un Ingress expose des Services HTTP/HTTPS vers l'extérieur.
- Il fonctionne via des règles de routage (host et path).
- Il nécessite un Ingress Controller pour être actif.
- Il ne parle jamais directement aux Pods.
- Il complète les Services pour l'exposition réseau.
