# Pods

## Qu'est-ce qu'un Pod ?

Le **Pod** est la plus petite unité de déploiement de Kubernetes.

Un Pod représente un environnement d'exécution dans lequel sont lancés un ou plusieurs conteneurs. Lorsqu'une application est exécutée dans Kubernetes, elle s'exécute toujours à l'intérieur d'un Pod.

Dans la majorité des cas, un Pod ne contient qu'un seul conteneur. Kubernetes autorise cependant plusieurs conteneurs au sein d'un même Pod lorsqu'ils doivent fonctionner très étroitement ensemble.

---

## Pourquoi un Pod et pas directement un conteneur ?

Contrairement à Docker, Kubernetes ne gère pas directement les conteneurs.

Le Pod fournit un environnement commun permettant à plusieurs conteneurs de partager :

- une même adresse IP ;
- les mêmes ports réseau ;
- un ou plusieurs volumes de stockage ;
- un même cycle de vie.

Le Pod constitue donc l'unité que Kubernetes crée, planifie, surveille et supprime.

---

## Cycle de vie d'un Pod

Un Pod suit généralement les étapes suivantes :

1. Création
2. Planification sur un Worker Node
3. Téléchargement des images
4. Démarrage des conteneurs
5. Exécution
6. Arrêt puis suppression

Un Pod est considéré comme **éphémère**.

Lorsqu'il disparaît (suppression, panne du nœud, etc.), Kubernetes ne le "répare" pas. Il en crée un nouveau si une ressource supérieure (par exemple un **Deployment**) indique qu'un Pod doit exister.

---

## Les phases d'un Pod

Kubernetes indique l'état global d'un Pod à l'aide d'une phase.

Les plus courantes sont :

| Phase | Description |
|--------|-------------|
| Pending | Le Pod est accepté mais n'est pas encore prêt à démarrer. |
| Running | Les conteneurs sont en cours d'exécution. |
| Succeeded | Tous les conteneurs se sont terminés avec succès. |
| Failed | Au moins un conteneur s'est terminé en erreur. |
| Unknown | Kubernetes ne peut pas déterminer l'état du Pod. |

Ces phases ne décrivent pas précisément l'état des conteneurs, mais donnent une vue d'ensemble.

---

## Un Pod n'est pas une machine virtuelle

Un Pod ne doit pas être considéré comme un serveur.

Il peut être recréé à tout moment avec :

- une nouvelle adresse IP ;
- un nouveau nom (selon la ressource qui le gère) ;
- un nouveau Worker Node.

Une application ne doit donc jamais dépendre directement de l'adresse IP d'un Pod.

Pour accéder à une application, on utilise généralement un **Service**.

---

## Les volumes (Stockage)

Par défaut, le système de fichiers d'un conteneur est éphémère : les données qu'il contient sont perdues lorsque le conteneur est recréé.

Pour partager des données entre conteneurs ou conserver des données au-delà du cycle de vie d'un conteneur, Kubernetes permet de monter des volumes dans un Pod.

Un volume est déclaré au niveau du Pod, puis monté dans un ou plusieurs conteneurs.

Cela permet notamment de :

- partager des fichiers entre plusieurs conteneurs d'un même Pod ;
- stocker des données temporaires ;
- monter des fichiers de configuration ;
- monter des Secrets ;
- utiliser un stockage persistant.

Dans l'exemple suivant, le volume `shared-data` est partagé entre les conteneurs du Pod.

```yaml
spec:
  volumes:
    - name: shared-data
      emptyDir: {}

  containers:
    - name: app
      image: my-app
      volumeMounts:
        - name: shared-data
          mountPath: /data

    - name: sidecar
      image: busybox
      volumeMounts:
        - name: shared-data
          mountPath: /shared
```

Le type de volume présenté ici (`emptyDir`) est créé lorsque le Pod démarre et supprimé lorsque celui-ci disparaît.

Les différents types de volumes, ainsi que le stockage persistant (`PersistentVolume`, `PersistentVolumeClaim`, `StorageClass`), sont détaillés dans la page **Volumes et stockage**.

---

## Les conteneurs multiples

Bien que le cas le plus courant soit un seul conteneur par Pod, plusieurs conteneurs peuvent partager un même Pod.

On parle souvent de **sidecar** lorsqu'un conteneur accompagne le conteneur principal pour lui fournir un service complémentaire.

Exemples :

- collecte des logs ;
- proxy réseau ;
- synchronisation de fichiers ;
- métriques.

Tous ces conteneurs partagent le même environnement réseau et les mêmes volumes.

---

## Exemple minimal

```yaml
apiVersion: v1
kind: Pod

metadata:
  name: nginx

spec:
  containers:
    - name: nginx
      image: nginx:latest
      ports:
        - containerPort: 80
```

Ce manifeste demande à Kubernetes de créer un Pod contenant un unique conteneur exécutant NGINX.

En pratique, il est rare de créer directement des Pods de cette manière.

---

## Pourquoi ne crée-t-on presque jamais un Pod directement ?

Créer un Pod fonctionne, mais présente un inconvénient majeur :

Si le Pod est supprimé, il ne sera pas recréé automatiquement.

Dans un environnement de production, les Pods sont généralement créés par des ressources de plus haut niveau :

- Deployment
- StatefulSet
- DaemonSet
- Job
- CronJob

Ces ressources sont responsables de maintenir le nombre de Pods souhaité.

---

## Commandes utiles

Lister les Pods :

```bash
kubectl get pods
```

Afficher des informations détaillées :

```bash
kubectl describe pod <nom-du-pod>
```

Afficher les logs :

```bash
kubectl logs <nom-du-pod>
```

Ouvrir un shell dans un conteneur :

```bash
kubectl exec -it <nom-du-pod> -- /bin/sh
```

Supprimer un Pod :

```bash
kubectl delete pod <nom-du-pod>
```

---

## Bonnes pratiques

- Un Pod exécute généralement une seule application.
- Ne pas stocker de données importantes dans le système de fichiers du Pod.
- Ne jamais utiliser directement l'adresse IP d'un Pod.
- Utiliser un Deployment pour les applications classiques.
- Utiliser des labels pour identifier les Pods.

---

## Ce qu'il faut retenir

- Le Pod est la plus petite unité déployable dans Kubernetes.
- Tous les conteneurs s'exécutent à l'intérieur d'un Pod.
- Les conteneurs d'un même Pod partagent le réseau et les volumes.
- Les Pods sont éphémères et peuvent être recréés à tout moment.
- Les Pods sont généralement créés et gérés par un Deployment, un StatefulSet ou une autre ressource Kubernetes.
