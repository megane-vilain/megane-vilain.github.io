# StatefulSets

## Qu'est-ce qu'un StatefulSet ?

Un **StatefulSet** est une ressource Kubernetes qui permet de gérer le cycle de vie d'applications ayant besoin d'une **identité stable** ou d'un **stockage persistant**.

Comme un **Deployment**, un StatefulSet crée et maintient un nombre de Pods souhaité. La différence est que chaque Pod possède une identité unique qui est conservée tout au long de son cycle de vie.

Les StatefulSets sont principalement utilisés pour les applications dites **stateful**, c'est-à-dire celles qui conservent des données ou dont chaque instance doit être identifiable.

---

## Pourquoi utiliser un StatefulSet ?

Un Deployment considère généralement que toutes les instances d'une application sont interchangeables.

Ce n'est pas le cas de certaines applications, par exemple :

- bases de données ;
- clusters de bases de données ;
- systèmes de messagerie ;
- applications nécessitant un stockage dédié à chaque instance.

Dans ces situations, chaque Pod doit conserver son identité et ses données même s'il est recréé.

Le StatefulSet répond à ce besoin.

---

## Ce qui différencie un StatefulSet d'un Deployment

Un StatefulSet garantit notamment :

- un nom stable pour chaque Pod ;
- un volume de stockage dédié à chaque Pod ;
- un ordre de création des Pods ;
- un ordre de suppression des Pods.

Par exemple, un StatefulSet nommé `database` avec trois réplicas créera les Pods :

```
database-0
database-1
database-2
```

Si `database-1` est recréé, il conservera le même nom et retrouvera son volume de stockage.

---

## Fonctionnement

Comme un Deployment, un StatefulSet s'appuie sur un contrôleur Kubernetes pour maintenir le nombre de Pods demandé.

Chaque Pod possède :

- un nom unique ;
- son propre volume persistant ;
- une identité réseau stable.

Contrairement à un Deployment, les Pods ne sont pas considérés comme interchangeables.

---

## Exemple minimal

```yaml
apiVersion: apps/v1
kind: StatefulSet

metadata:
  name: database

spec:
  serviceName: database
  replicas: 3

  selector:
    matchLabels:
      app: database

  template:
    metadata:
      labels:
        app: database

    spec:
      containers:
        - name: postgres
          image: postgres:17
          ports:
            - containerPort: 5432

  volumeClaimTemplates:
    - metadata:
        name: data

      spec:
        accessModes:
          - ReadWriteOnce

        resources:
          requests:
            storage: 10Gi
```

Ce manifeste demande à Kubernetes de créer trois Pods :

- `database-0`
- `database-1`
- `database-2`

Chaque Pod recevra automatiquement son propre volume persistant.

---

## Les principaux champs

### serviceName

Chaque StatefulSet doit être associé à un **Service** (généralement un *Headless Service*).

Ce Service permet aux Pods d'obtenir une identité réseau stable.

```yaml
serviceName: database
```

---

### replicas

Détermine le nombre de Pods souhaités.

```yaml
replicas: 3
```

Chaque Pod possède sa propre identité.

---

### template

Le template décrit le Pod qui sera créé.

Comme pour un Deployment, toute modification du template entraînera une mise à jour des Pods.

---

### volumeClaimTemplates

Cette section permet de créer automatiquement un **PersistentVolumeClaim** pour chaque Pod.

Par exemple :

```
database-0 → PVC data-database-0
database-1 → PVC data-database-1
database-2 → PVC data-database-2
```

Chaque Pod dispose ainsi de son propre espace de stockage.

---

## Création et suppression des Pods

Contrairement aux Deployments, les Pods d'un StatefulSet sont créés dans un ordre précis.

Par exemple, avec trois réplicas :

```
database-0
      ↓
database-1
      ↓
database-2
```

Kubernetes attend généralement qu'un Pod soit prêt avant de créer le suivant.

La suppression suit l'ordre inverse :

```
database-2
      ↓
database-1
      ↓
database-0
```

Cela facilite le fonctionnement de nombreuses applications distribuées.

---

## Mise à jour

Les StatefulSets prennent également en charge les mises à jour.

Par défaut, les Pods sont mis à jour un par un afin de limiter les interruptions de service.

---

## Quand utiliser un StatefulSet ?

Le StatefulSet est adapté aux applications ayant besoin :

- d'un stockage persistant par instance ;
- d'un nom stable ;
- d'une identité réseau stable ;
- d'un ordre de démarrage ou d'arrêt.

Exemples :

- bases de données ;
- clusters PostgreSQL ;
- clusters MariaDB ;
- clusters Kafka ;
- clusters Elasticsearch.

---

## Quand ne pas utiliser un StatefulSet ?

Un StatefulSet n'est pas nécessaire pour une application dont toutes les instances sont interchangeables.

Pour une API REST, un frontend web ou un microservice sans état, un **Deployment** est généralement plus adapté.

---

## Commandes utiles

Lister les StatefulSets :

```bash
kubectl get statefulsets
```

Afficher les détails :

```bash
kubectl describe statefulset <nom>
```

Modifier le nombre de réplicas :

```bash
kubectl scale statefulset <nom> --replicas=5
```

Afficher l'état du déploiement :

```bash
kubectl rollout status statefulset <nom>
```

---

## Bonnes pratiques

- Utiliser un StatefulSet uniquement lorsqu'une identité stable est nécessaire.
- Prévoir un stockage persistant adapté à l'application.
- Ne pas supposer que tous les Pods sont interchangeables.
- Associer le StatefulSet à un Service adapté.

---

## Ce qu'il faut retenir

- Un StatefulSet gère les applications ayant un **état** (stateful).
- Chaque Pod possède un nom stable et une identité propre.
- Chaque Pod dispose généralement de son propre volume persistant.
- Les Pods sont créés et supprimés dans un ordre déterminé.
- Pour les applications sans état, un Deployment reste le choix recommandé.
