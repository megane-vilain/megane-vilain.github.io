# Secrets

## Qu'est-ce qu'un Secret ?

Un **Secret** est une ressource Kubernetes permettant de stocker des données sensibles sous forme de paires **clé/valeur**.

Comme une **ConfigMap**, un Secret peut être utilisé par les Pods sous forme de variables d'environnement ou de fichiers montés dans un volume.

La différence est que les Secrets sont destinés aux informations qui ne doivent pas être exposées, telles que les mots de passe, les clés API ou les certificats.

---

## Pourquoi utiliser un Secret ?

Il est déconseillé d'intégrer des informations sensibles directement :

- dans le code source ;
- dans l'image du conteneur ;
- dans les manifestes Kubernetes.

Les Secrets permettent de séparer les données sensibles de l'application et de les gérer indépendamment.

---

## Que peut contenir un Secret ?

Un Secret peut contenir par exemple :

- des mots de passe ;
- des clés API ;
- des tokens d'authentification ;
- des certificats TLS ;
- des clés privées ;
- des identifiants de connexion.

Les données de configuration qui ne sont pas sensibles doivent, elles, être stockées dans une **ConfigMap**.

---

## Comment sont stockées les données ?

Les valeurs d'un Secret sont généralement écrites en **Base64** dans les manifestes Kubernetes.

Exemple :

```yaml
apiVersion: v1
kind: Secret

metadata:
  name: database-secret

type: Opaque

data:
  username: YWRtaW4=
  password: c2VjcmV0
```

Il est important de noter que le **Base64 n'est pas un mécanisme de chiffrement**.

Il s'agit simplement d'un format d'encodage utilisé pour représenter des données binaires sous forme de texte.

La protection réelle des Secrets dépend de la configuration du cluster Kubernetes (chiffrement dans `etcd`, contrôle des accès via RBAC, etc.).

---

## Créer un Secret avec `stringData`

Lorsqu'on écrit un manifeste à la main, il est souvent plus pratique d'utiliser le champ `stringData`.

```yaml
apiVersion: v1
kind: Secret

metadata:
  name: database-secret

type: Opaque

stringData:
  username: admin
  password: secret
```

Kubernetes se charge automatiquement d'encoder les valeurs en Base64 lors de la création du Secret.

Cette méthode est généralement plus lisible que l'utilisation directe du champ `data`.

---

## Utiliser un Secret comme variable d'environnement

Les valeurs d'un Secret peuvent être injectées dans un conteneur sous forme de variables d'environnement.

```yaml
env:
  - name: DB_USERNAME
    valueFrom:
      secretKeyRef:
        name: database-secret
        key: username

  - name: DB_PASSWORD
    valueFrom:
      secretKeyRef:
        name: database-secret
        key: password
```

L'application peut ensuite accéder à ces valeurs comme à n'importe quelle variable d'environnement.

---

## Monter un Secret comme volume

Un Secret peut également être monté dans un volume.

Chaque clé devient alors un fichier.

```yaml
volumes:
  - name: secret-volume
    secret:
      secretName: database-secret

containers:
  - name: app
    volumeMounts:
      - name: secret-volume
        mountPath: /etc/secrets
        readOnly: true
```

Le conteneur retrouvera par exemple :

```
/etc/secrets/username
/etc/secrets/password
```

Cette méthode est particulièrement adaptée aux certificats ou aux clés privées.

---

## Les différents types de Secrets

Le type le plus courant est :

```yaml
type: Opaque
```

Il permet de stocker n'importe quelle donnée.

Kubernetes fournit également plusieurs types spécialisés, notamment :

| Type | Utilisation |
|------|-------------|
| `Opaque` | Données arbitraires |
| `kubernetes.io/tls` | Certificats TLS |
| `kubernetes.io/dockerconfigjson` | Authentification auprès d'un registre de conteneurs |
| `kubernetes.io/basic-auth` | Nom d'utilisateur et mot de passe |
| `kubernetes.io/ssh-auth` | Clés SSH |

Dans la plupart des cas, les Secrets de type `Opaque` sont suffisants.

---

## Modifier un Secret

Comme les ConfigMaps, un Secret peut être modifié.

Cependant, selon la manière dont il est utilisé :

- une variable d'environnement ne sera généralement prise en compte qu'au redémarrage du Pod ;
- un Secret monté comme volume peut être mis à jour automatiquement selon l'application.

Dans de nombreux cas, un redémarrage des Pods reste nécessaire.

---

## Ce que les Secrets ne garantissent pas

L'utilisation d'un Secret ne signifie pas que les données sont automatiquement chiffrées ou totalement protégées.

Par exemple :

- les valeurs sont simplement encodées en Base64 dans les manifestes ;
- un utilisateur disposant des droits RBAC appropriés peut lire les Secrets ;
- un Pod ayant accès à un Secret peut en lire le contenu.

Les Secrets améliorent l'organisation et la gestion des données sensibles, mais leur niveau de protection dépend également de la configuration du cluster.

---

## Bonnes pratiques

- Utiliser un Secret pour toutes les données sensibles.
- Préférer `stringData` lors de l'écriture des manifestes.
- Donner accès aux Secrets uniquement aux Pods qui en ont besoin.
- Limiter les permissions RBAC permettant de consulter les Secrets.
- Éviter de stocker les manifestes contenant des valeurs réelles dans un dépôt Git.
- Utiliser des solutions de gestion de secrets (Vault, External Secrets Operator, Sealed Secrets...) lorsque cela est nécessaire.

---

## Commandes utiles

Lister les Secrets :

```bash
kubectl get secrets
```

Afficher les détails :

```bash
kubectl describe secret <nom>
```

Afficher le manifeste :

```bash
kubectl get secret <nom> -o yaml
```

Supprimer un Secret :

```bash
kubectl delete secret <nom>
```

Créer un Secret depuis la ligne de commande :

```bash
kubectl create secret generic database-secret \
  --from-literal=username=admin \
  --from-literal=password=secret
```

---

## ConfigMap ou Secret ?

| ConfigMap | Secret |
|-----------|--------|
| Configuration non sensible | Données sensibles |
| URL de services | Mots de passe |
| Niveau de logs | Clés API |
| Options d'application | Tokens |
| Paramètres métier | Certificats |

Les deux ressources sont utilisées de manière similaire dans les Pods. Le choix dépend principalement de la sensibilité des données.

---

## Ce qu'il faut retenir

- Un Secret permet de stocker des données sensibles.
- Il peut être utilisé comme variable d'environnement ou comme volume.
- Les valeurs sont souvent représentées en Base64, qui n'est **pas** un mécanisme de chiffrement.
- Les permissions d'accès aux Secrets doivent être limitées.
- Pour les données non sensibles, il est préférable d'utiliser une ConfigMap.
