# Paquets `-dev` Debian et intégration CMake (`find_package`)

Cette documentation explique :

- à quoi servent les paquets `-dev` sous Debian,
- comment générer les fichiers CMake de configuration (`Config.cmake`, `ConfigVersion.cmake`, `Targets.cmake`…),
- comment exporter vos targets pour permettre l’utilisation de `find_package()`.

---

## TL;DR

### Côté bibliothèque (producteur)

1. Créer une cible de bibliothèque :

   ```cmake
   add_library(Foo SHARED src/foo.c)
   ```

2. Déclarer les includes avec `BUILD_INTERFACE` / `INSTALL_INTERFACE` :

   ```cmake
   target_include_directories(Foo
     PUBLIC
       $<BUILD_INTERFACE:${CMAKE_CURRENT_SOURCE_DIR}/include>
       $<INSTALL_INTERFACE:${CMAKE_INSTALL_INCLUDEDIR}/Foo>
   )
   ```

3. Installer la bibliothèque et exporter les targets :

   ```cmake
   install(TARGETS Foo
     EXPORT FooTargets
     PUBLIC_HEADER DESTINATION ${CMAKE_INSTALL_INCLUDEDIR}/Foo
   )

   install(EXPORT FooTargets
     NAMESPACE Foo::
     FILE FooTargets.cmake
     DESTINATION ${CMAKE_INSTALL_DATAROOTDIR}/Foo/cmake
   )
   ```

4. Générer les fichiers de configuration :

   ```cmake
   configure_package_config_file(
     cmake/Config.cmake.in
     ${CMAKE_CURRENT_BINARY_DIR}/FooConfig.cmake
     INSTALL_DESTINATION ${CMAKE_INSTALL_DATAROOTDIR}/Foo/cmake
   )

   write_basic_package_version_file(
     ${CMAKE_CURRENT_BINARY_DIR}/FooConfigVersion.cmake
     VERSION ${PROJECT_VERSION}
     COMPATIBILITY SameMajorVersion
   )

   install(FILES
     ${CMAKE_CURRENT_BINARY_DIR}/FooConfig.cmake
     ${CMAKE_CURRENT_BINARY_DIR}/FooConfigVersion.cmake
     DESTINATION ${CMAKE_INSTALL_DATAROOTDIR}/Foo/cmake
   )
   ```

5. Mettre ces fichiers (headers + `FooConfig.cmake`, `FooConfigVersion.cmake`, `FooTargets.cmake`) dans le paquet Debian `libfoo-dev`.

---

### Côté utilisateur (consommateur)

1. Installer le paquet de développement :

   ```bash
   apt install libfoo-dev
   ```

2. Dans le projet CMake :

   ```cmake
   cmake_minimum_required(VERSION 3.16)
   project(MyApp C)

   find_package(Foo REQUIRED)
   add_executable(my_app main.c)
   target_link_libraries(my_app PRIVATE Foo::Foo)
   ```

## 1. Paquets `-dev` sous Debian : à quoi ça sert ?

Sur Debian (et dérivés), les paquets `-dev` contiennent tout ce qui est nécessaire pour **développer** avec une bibliothèque :

- fichiers d’en-tête (`.h`, `.hpp`),
- fichiers de configuration CMake (`*Config.cmake`, `*Targets.cmake`, …),

Exemples :

- `libssl-dev` pour OpenSSL,
- `libcurl4-openssl-dev` pour libcurl,
- `libfoo-dev` pour une bibliothèque fictive `libfoo`.

Sans ces paquets `-dev`, CMake a souvent du mal à trouver correctement la bibliothèque via `find_package()`.

---

## 2. Utiliser `find_package()` avec les paquets `-dev`

CMake fournit deux grands modes pour `find_package()` :

1. **Mode “Config”**
   CMake cherche un fichier du type :
   - `FooConfig.cmake`
   - ou `foo-config.cmake`

   Ce fichier est généralement fourni par le paquet `libfoo-dev` dans un dossier comme :
   - `/usr/lib/cmake/Foo/FooConfig.cmake`
   - ou `/usr/lib/x86_64-linux-gnu/cmake/Foo/FooConfig.cmake`

2. **Mode “Module”**
   CMake cherche un module dans son propre répertoire :
   - `FindFoo.cmake`

   Ce mode est utilisé quand la bibliothèque n’installe pas elle-même un `FooConfig.cmake`.

### Exemple d’utilisation côté consommateur

Dans le projet qui **utilise** la bibliothèque :

```cmake
cmake_minimum_required(VERSION 3.16)
project(MyApp C)

find_package(Foo REQUIRED)  # Foo vient par exemple de libfoo-dev

add_executable(my_app main.c)
target_link_libraries(my_app PRIVATE Foo::Foo) # target exportée par Foo
```

Pour que cela fonctionne :

- le paquet `libfoo-dev` doit être installé,
- il doit contenir le fichier `FooConfig.cmake` (ou équivalent),
- ce fichier doit définir une ou plusieurs targets (`Foo::Foo`, etc.).

---

## 3. Côté bibliothèque : générer les fichiers de config CMake

Si vous développez une bibliothèque C que vous voulez **rendre trouvable** via `find_package()`, vous devez :

1. Exporter vos targets dans un fichier `Targets.cmake`.
2. Générer un fichier `Config.cmake`.
3. Optionnellement, un fichier `ConfigVersion.cmake`.

### 3.1. Structure minimale dans `CMakeLists.txt`

Exemple pour une bibliothèque simple `foo` :

```cmake
### 3.1 Structure minimale `CMakeLists.txt` (avec installation des headers via PUBLIC_HEADER)

cmake_minimum_required(VERSION 3.16)
project(Foo VERSION 1.0 LANGUAGES C)

include(GNUInstallDirs)
include(CMakePackageConfigHelpers)

# --- 1) Déclarer les headers publics ---
# Liste des headers publics de la bibliothèque (API exposée)
set(HEADERS    include/foo.h)

# --- 2) Construire la bibliothèque ---
add_library(${PROJECT_NAME} SHARED
    src/foo.c
)

# Déclarer où trouver les headers :
# - pendant la compilation (dans l'arborescence du projet)
# - après l'installation (installés sous include/${PROJECT_NAME})
target_include_directories(${PROJECT_NAME}
    PUBLIC
        $<BUILD_INTERFACE:${CMAKE_CURRENT_SOURCE_DIR}/include>
        $<INSTALL_INTERFACE:${CMAKE_INSTALL_INCLUDEDIR}/${PROJECT_NAME}>
)

# Associer la liste de headers publics à la cible via PUBLIC_HEADER
set_target_properties(${PROJECT_NAME} PROPERTIES PUBLIC_HEADER "${HEADERS}")

# --- 3) Installer la bibliothèque et ses headers ---
install(TARGETS ${PROJECT_NAME}
    EXPORT "${PROJECT_NAME}Targets"
    LIBRARY       DESTINATION ${CMAKE_INSTALL_LIBDIR} COMPONENT Libraries
    ARCHIVE       DESTINATION ${CMAKE_INSTALL_LIBDIR} COMPONENT Libraries
    RUNTIME       DESTINATION ${CMAKE_INSTALL_BINDIR} COMPONENT Binaries
    PUBLIC_HEADER DESTINATION ${CMAKE_INSTALL_INCLUDEDIR}/${PROJECT_NAME} COMPONENT Headers
)

# --- 5) Exporter les targets vers un fichier FooTargets.cmake ---
install(EXPORT "${PROJECT_NAME}Targets"
    NAMESPACE Foo::
    FILE ${PROJECT_NAME}Targets.cmake
    DESTINATION ${CMAKE_INSTALL_DATAROOTDIR}/${PROJECT_NAME}/cmake
)

# --- 6) Générer FooConfig.cmake (template -> fichier final) ---
configure_package_config_file(
    ${CMAKE_CURRENT_SOURCE_DIR}/cmake/Config.cmake.in
    ${CMAKE_CURRENT_BINARY_DIR}/${PROJECT_NAME}Config.cmake
    INSTALL_DESTINATION ${CMAKE_INSTALL_DATAROOTDIR}/${PROJECT_NAME}/cmake
)

# --- 7) Générer FooConfigVersion.cmake (pour find_package(Foo X.Y)) ---
write_basic_package_version_file(
    ${CMAKE_CURRENT_BINARY_DIR}/${PROJECT_NAME}ConfigVersion.cmake
    VERSION ${PROJECT_VERSION}
    COMPATIBILITY SameMajorVersion
)

# --- 8) Installer les fichier FooConfig.cmake et FooConfigVersion.cmake
install(FILES
  ${CMAKE_CURRENT_BINARY_DIR}/${PROJECT_NAME}Config.cmake
    ${CMAKE_CURRENT_BINARY_DIR}/${PROJECT_NAME}ConfigVersion.cmake
    DESTINATION ${CMAKE_INSTALL_DATAROOTDIR}/${PROJECT_NAME}/cmake
)
```

### 3.1.1. Attention aux chemins relatifs dans `target_include_directories()` (export de cibles)

Lorsque vous exportez vos cibles (via `install(EXPORT ...)` pour générer `FooTargets.cmake`),
les répertoires d’includes déclarés dans `target_include_directories()` deviennent partie
intégrante de l’interface publique de la bibliothèque. Ils seront réutilisés dans les projets
consommateurs via `find_package(Foo)`.

Il ne faut pas utiliser des chemins relatifs à l’arborescence des sources,
par exemple :

```cmake
# À éviter
target_include_directories(foo PUBLIC src/include)
```

Une fois la cible exportée et réutilisée dans un autre projet, ce chemin `src/include` n’a
plus de sens : le projet consommateur n’a pas accès à l’arborescence des sources de `Foo` et
CMake n’a aucun moyen de deviner où se trouve ce `src/include`.

C’est pour cela que l’exemple utilise les générateurs `BUILD_INTERFACE` et `INSTALL_INTERFACE` :

```cmake
target_include_directories(${PROJECT_NAME}
    PUBLIC
        $<BUILD_INTERFACE:${CMAKE_CURRENT_SOURCE_DIR}/include>
        $<INSTALL_INTERFACE:${CMAKE_INSTALL_INCLUDEDIR}/${PROJECT_NAME}>
)
```

- `BUILD_INTERFACE` est utilisé **pendant la compilation du projet de la bibliothèque**.
  Ici, `CMAKE_CURRENT_SOURCE_DIR` pointe vers le répertoire de la bibliothèque dans l’arborescence
  des sources, donc `CMAKE_CURRENT_SOURCE_DIR}/include` est un chemin valide uniquement lors du
  build de la bibliothèque elle-même.

- `INSTALL_INTERFACE` est utilisé **après installation**, dans les projets qui consomment la
  bibliothèque via `find_package(Foo)`. Le chemin `${CMAKE_INSTALL_INCLUDEDIR}/${PROJECT_NAME}` est interprété
  relativement au préfixe d’installation (`CMAKE_INSTALL_PREFIX`) et correspond au chemin où
  les headers publics ont été installés via `PUBLIC_HEADER DESTINATION ${CMAKE_INSTALL_INCLUDEDIR}/${PROJECT_NAME}`

### 3.2. Génération `FooConfig.cmake.in`

Il est nécessaire d'avoir un fichier `Config.cmake.in` qui est un **template** placé dans le répertoire `cmake/` à la racine du projet :

```
/project-root
  /src
  /include
  /cmake
    Config.cmake.in
  CMakeLists.txt
```

CMake va le transformer en `FooConfig.cmake` lors de l’installation, grâce à `configure_package_config_file()`.

#### 3.2.1. Exemple minimal adapté

Voici un exemple de `cmake/FooConfig.cmake.in` adapté à ce contexte :

```cmake
@PACKAGE_INIT@

# Import des targets exportées par le projet (Foo::Foo, etc.)
include("${CMAKE_CURRENT_LIST_DIR}/@PROJECT_NAME@Targets.cmake")

# Vérifie que tous les composants requis sont bien disponibles
check_required_components("@PROJECT_NAME@")
```

Points importants :

- `@PACKAGE_INIT@` : insère du code standard généré par CMake (gestion des chemins, vérifications, etc.).
- `include("${CMAKE_CURRENT_LIST_DIR}/@PROJECT_NAME@Targets.cmake")` : charge les targets exportées (par ex. `Foo::Foo`), définies dans le fichier `FooTargets.cmake` généré par `install(EXPORT ...)`.
- `check_required_components("@PROJECT_NAME@")` : utile si ton package supporte la notion de **composants** (ex. `find_package(Foo COMPONENTS Core Net)`).

Si ton package ne gère pas de composants, `check_required_components("@PROJECT_NAME@")` ne fera pas grand-chose, mais sa présence ne gêne pas.

---

#### 3.2.2. Gestion des dépendances de la librairie

Le même `Config.cmake` peut aussi devoir **déclarer des dépendances** vers d’autres bibliothèques CMake. Il sera nécessaire d'utiliser la fonction `find_dependency()`.

Cela se produit dans les cas suivants :

1. Ta bibliothèque `Foo` **linke** une autre bibliothèque (par exemple `Zlib`, `OpenSSL`, `Threads`, etc.).
2. Tu exposes cette autre bibliothèque dans l’interface de `Foo` :
   - via `target_link_libraries(foo PUBLIC ZLIB::ZLIB)` par exemple,
   - ou via des headers qui nécessitent ces dépendances.

Dans ce cas, tout projet qui fait :

```cmake
find_package(Foo REQUIRED)
target_link_libraries(my_app PRIVATE Foo::Foo)
```

doit aussi avoir accès aux dépendances de `Foo`.
Pour que cela soit automatique et propre, on utilise `find_dependency()` dans `FooConfig.cmake.in`.

---

#### 3.2.3. Exemple avec `find_dependency()`

Supposons que ta bibliothèque `Foo` dépende de `Zlib` et d’`OpenSSL`. Tu peux écrire :

```cmake
@PACKAGE_INIT@

# Import de la fonction find_dependency() fournie par CMakePackageConfigHelpers
include(CMakeFindDependencyMacro)

# Déclaration des dépendances de Foo
find_dependency(ZLIB)
find_dependency(OpenSSL)


# Import des targets exportées par Foo
include("${CMAKE_CURRENT_LIST_DIR}/@PROJECT_NAME@Targets.cmake")

check_required_components("@PROJECT_NAME@")
```

Explications :

- `include(CMakeFindDependencyMacro)` : rend disponible la macro `find_dependency()`.
- `find_dependency(ZLIB)` : équivalent à un `find_package(ZLIB REQUIRED)` mais adapté pour être utilisé dans un fichier `Config.cmake`.
- `find_dependency(OpenSSL)` : idem pour OpenSSL.
- Si ces dépendances ne sont pas trouvées, `find_package(Foo)` échouera, ce qui évite d’avoir un projet partiellement configuré et qui ne compile pas.

---

#### 3.2.4. Quand ajouter des `find_dependency()` ?

Tu ajoutes des `find_dependency()` dans ton `FooConfig.cmake.in` lorsque :

- ta cible `Foo::Foo` est **PUBLIC**-linkée avec une autre librairie CMake :

  ```cmake
  target_link_libraries(foo
      PUBLIC ZLIB::ZLIB
      PUBLIC OpenSSL::SSL
  )
  ```

- ou lorsque l’utilisation normale de ta librairie **requiert** une autre librairie (tu ne veux pas que `Foo` soit considéré comme « trouvé » si sa dépendance ne l’est pas).

En revanche, tu peux éviter `find_dependency()` si :

- la dépendance est totalement **interne** à ton projet (linkage PRIVATE) et n’a aucun impact sur les consommateurs :

  ```cmake
  target_link_libraries(foo
      PRIVATE SomeInternalLib
  )
  ```

- ou si c’est une dépendance optionnelle, et que `Foo` reste utilisable dans un mode dégradé sans elle (dans ce cas, tu gères plutôt des options/variables CMake côté utilisateur).

---

#### 3.2.5. Résumé pour `Config.cmake.in`

Pour une bibliothèque classique installée via un paquet `-dev`, un bon template de base est :

```cmake
@PACKAGE_INIT@

# Si la bibliothèque dépend d'autres packages CMake :
include(CMakeFindDependencyMacro)
# Exemple :
# find_dependency(ZLIB)
# find_dependency(OpenSSL)

# Chargement des targets exportées
include("${CMAKE_CURRENT_LIST_DIR}/@PROJECT_NAME@Targets.cmake")

# Vérification des composants (utile si tu utilises COMPONENTS dans find_package)
check_required_components("@PROJECT_NAME@")
```

Avec ce template :

- `find_package(Foo)` fonctionnera correctement via le paquet `libfoo-dev`,
- les targets `Foo::Foo` seront disponibles,
- les dépendances nécessaires seront automatiquement recherchées via `find_dependency()`,
- l’API CMake de ta bibliothèque sera propre et prévisible.

## 4. Quels fichiers sont générés dans le paquet `-dev` ?

Lorsque vous installez votre bibliothèque via CMake, le paquet `-dev` Debian doit inclure les fichiers suivants :

### 4.1. FooConfig.cmake

- Fichier principal de configuration.
- Permet à `find_package(Foo)` de localiser :
  - les targets,
  - les chemins d’includes,
  - éventuellement des options (features, composants…).

### 4.2. FooConfigVersion.cmake

- Permet à `find_package()` de vérifier la version.
- Utilisé lorsqu’on écrit : `find_package(Foo 1.2 REQUIRED)`.

Généré automatiquement par `write_basic_package_version_file`.
Il contient les informations nécessaires pour vérifier que la version de la bibliothèque est compatible avec celle demandée.

### 4.3. FooTargets.cmake

- Contient la **définition des targets** exportées :
  - `Foo::Foo`
  - éventuellement d’autres (`Foo::Extra`, etc.).
- Ces targets sont utilisables directement via `target_link_libraries()` dans les projets consommateurs.

Généré via :

```cmake
install(
        EXPORT "${PROJECT_NAME}Targets"
        FILE ${PROJECT_NAME}Targets.cmake
        NAMESPACE Foo::
        DESTINATION ${CMAKE_INSTALL_DATAROOTDIR}/${PROJECT_NAME}/cmake
)
```

### 4.4. En-têtes et autres fichiers

En plus des fichiers CMake, le paquet `libfoo-dev` doit contenir :

- Les fichiers header de la librarie

## 5. Packaging Debian : où CMake va chercher les fichiers de configuration ?

Pour que `find_package(Foo)` fonctionne, CMake recherche par défaut les fichiers
`FooConfig.cmake`, `FooConfigVersion.cmake` et `FooTargets.cmake` dans un ensemble de répertoires « standards », déterminés par la plateforme et la configuration de CMake.
Sur Debian, cela inclut typiquement des chemins comme :

- `/usr/lib/cmake/Foo/`
- `/usr/lib/x86_64-linux-gnu/cmake/Foo/`

Mais il est possible d’influencer ces emplacements :

- **`CMAKE_SYSTEM_PREFIX_PATH`** permet d’ajouter ou modifier des préfixes système que CMake considère comme standards pour rechercher les paquets (utile pour des répertoires système supplémentaires ou non conventionnels).
- **`CMAKE_INSTALL_PREFIX`** définit le préfixe d’installation de ton projet (par défaut `/usr/local`), ce qui conditionne là où tes propres fichiers `FooConfig.cmake` & co seront installés. Ces chemins sont ensuite pris en compte par `find_package` lorsque tu utilises ce préfixe.

---

## 6. Résumé pratique

- Installer le paquet `libfoo-dev` permet d’avoir :
  - les headers,
  - les fichiers `FooConfig.cmake`, `FooConfigVersion.cmake`, `FooTargets.cmake`.

- Dans le projet utilisateur :

  ```cmake
  find_package(Foo REQUIRED)
  target_link_libraries(my_app PRIVATE Foo::Foo)
  ```

- Côté bibliothèque :
  - utiliser `install(TARGETS ...)` + `install(EXPORT ...)` pour générer `FooTargets.cmake`,
  - utiliser `configure_package_config_file()` pour `FooConfig.cmake`,
  - utiliser `write_basic_package_version_file()` pour `FooConfigVersion.cmake`,
  - installer ces fichiers dans un répertoire `lib/cmake/Foo` intégré dans le paquet `-dev`.

Cette approche permet une intégration propre avec CMake et un packaging Debian conforme, facilitant la vie des utilisateurs de la librairie.
